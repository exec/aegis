# RTL8169/8168 Driver Implementation Plan

**Goal:** Add a polling-mode `kernel/drivers/rtl8169.c` that registers an `eth0` `netdev_t` for Realtek RTL8168 (vendor `10ec`, device `8168`) and lets DHCP/ARP/ICMP/TCP work end-to-end on bare-metal AMD hardware (initial target: ASUS EQR with dual RTL8168, validated via VFIO PCI passthrough during dev).

**Architecture:** Mirrors `virtio_net.c`. Static single-instance state. MMIO via BAR2 mapped uncached through `kva_alloc_pages` + `vmm_map_page`. Bus master + memory enable via PCI config space, wake from D3 via PMCSR. 256-entry RX and TX rings of 16-byte descriptors, single-segment per descriptor, no scatter-gather, no checksum offload. RX buffers are full pages (4096), well above the 1518-byte Ethernet max. TX uses bounce buffers, one page each. Polling drains RX from `netdev_poll_all()` (PIT 100Hz). No IRQ. No PHY init magic — VFIO and bare-metal both leave the PHY in a usable state after BIOS init.

**Tech stack:** C, kernel headers (`pcie.h`, `kva.h`, `vmm.h`, `pmm.h`, `printk.h`, `arch.h` for `arch_wmb`), `kernel/net/netdev.h`.

## Register cheat sheet (8168 family)

```
0x00  MAC0      32-bit   low 4 bytes of MAC address (read-only after reset)
0x04  MAC4      32-bit   high 2 bytes of MAC address (in low 16 bits)
0x20  TxLow     32-bit   TX descriptor ring physical base, low
0x24  TxHigh    32-bit   TX descriptor ring physical base, high
0x37  ChipCmd    8-bit   bit 4=Reset, bit 3=RxEnb, bit 2=TxEnb
0x38  TxPoll     8-bit   bit 6 (NPQ)=kick normal-priority TX
0x3C  IntrMask  16-bit   IRQ mask (we set 0)
0x3E  IntrStatus 16-bit  IRQ status (write to clear)
0x40  TxConfig  32-bit   TX DMA burst + interframe gap
0x44  RxConfig  32-bit   RX accept mask + DMA burst
0x50  Cfg9346    8-bit   0xC0=Unlock, 0x00=Lock (gate config writes)
0xDA  RxMaxSize 16-bit   max RX frame size we accept
0xE0  CPlusCmd  16-bit   C+ extras (RX checksum offload, etc.)
0xE4  RxLow     32-bit   RX descriptor ring physical base, low
0xE8  RxHigh    32-bit   RX descriptor ring physical base, high
```

Descriptor (16 bytes, little-endian):
```
opts1: bit 31 = DescOwn (1=NIC owns, 0=driver owns)
       bit 30 = RingEnd
       bit 29 = FirstFrag  (TX only)
       bit 28 = LastFrag   (TX only)
       bits 13:0 = length (RX rcv length / TX frag length)
opts2: 0 (no VLAN, no checksum offload)
addr:  64-bit physical buffer address
```

ChipCmd values:
```
0x10 CmdReset
0x08 CmdRxEnb
0x04 CmdTxEnb
```

PCI config:
- `0x04` Command register: set bits `0x06` = Memory + BusMaster
- `0x44` PMCSR (offset depends on PM cap location, typically `0x44` for these chips): low 2 bits = power state, write 0 for D0

## Task 1: Register definitions header

```c
#ifndef RTL8169_H
#define RTL8169_H
void rtl8169_init(void);
#endif
```

That's it — register constants stay private to the .c file.

## Task 2: Driver skeleton — find device, map BAR2, dump info

Implement `rtl8169_init` up to the point where we've:
1. Found the device via `pcie_get_devices()` matching vendor `0x10EC`, device `0x8168`
2. Logged `[RTL8169] OK: found at <bus>:<dev>.<fn>`
3. Set PCI command register bits `0x06` (Memory + BusMaster)
4. Wrote `0x00` to PMCSR + 4 to wake to D0 (best-effort, no PM cap walk yet — most chips wake on first MMIO access too, but this guards against being stuck in D3)
5. Mapped BAR2 (`d->bar[2] & ~0xFFF`) as 1 page of uncached MMIO via `map_bar_region` helper (cribbed from virtio_net.c)
6. Read MAC0/MAC4, logged `[RTL8169] OK: mac=xx:xx:xx:xx:xx:xx`
7. Returned without registering a netdev yet

**Test:** Boot Aegis under VFIO passthrough on EQR, see:
```
[PCIE] found 10ec:8168 class=2 at 0:3.0
[RTL8169] OK: found at 0:3.0
[RTL8169] OK: mac=78:55:36:01:ff:73
```
The MAC must match the host's `enp2s0` MAC (`78:55:36:01:ff:73`). If MAC reads as `00:00:00:00:00:00` or `ff:ff:ff:ff:ff:ff`, BAR mapping is wrong.

## Task 3: Reset chip

Add a reset step before reading MAC:
```c
mmio_write8(ChipCmd, CmdReset);
for (int spin = 0; spin < 100; spin++) {
    if ((mmio_read8(ChipCmd) & CmdReset) == 0) break;
    busy_wait_us(100);
}
```
If reset bit doesn't clear within ~10ms, log `[RTL8169] FAIL: reset stuck` and return.

**Test:** Same as Task 2, but reset must complete cleanly. MAC should still read correctly post-reset (chip reloads from eFuse).

## Task 4: Allocate descriptor rings + RX buffers

Allocate:
- 1 page for the TX descriptor ring (256 × 16 = 4096 bytes — exactly one page)
- 1 page for the RX descriptor ring (same)
- 256 pages for RX data buffers (use `alloc_dma_page` from a private helper similar to virtio_net.c)
- 256 pages for TX bounce buffers

Initialize all 256 RX descriptors:
```c
rx_desc[i].opts1 = DescOwn | RX_BUF_SIZE;   /* device owns it, ready to fill */
rx_desc[i].opts2 = 0;
rx_desc[i].addr  = rx_buf_pa[i];
```
Set `RingEnd` on `rx_desc[255]`. Same for TX, but `opts1 = 0` (driver owns) and last entry gets `RingEnd`.

Use `RX_BUF_SIZE = 2048`. The buffer is 4096 bytes (one page) so 2048 leaves a fat margin.

Pseudocode helper to keep init readable:
```c
static void alloc_dma_page(uint64_t *pa_out, uintptr_t *va_out);  /* same as virtio_net */
```

**Test:** After Task 4 the rings exist but RX/TX is still off. Re-run, look for no panic, no MMIO complaints.

## Task 5: Configure registers + enable RX/TX

In order:
```c
mmio_write8 (Cfg9346, 0xC0);                /* Unlock */
mmio_write32(TxConfig, (3 << 24) | (3 << 8)); /* Tx DMA burst 1024, IFG */
mmio_write32(RxConfig, (7 << 13) | (7 << 8) /* fifo, dma */
                       | (1 << 1) | (1 << 2) | (1 << 3));
                       /* AcceptMyPhys | AcceptBroadcast | AcceptMulticast */
mmio_write16(RxMaxSize, 1538);
mmio_write16(CPlusCmd, mmio_read16(CPlusCmd) | (1 << 3)); /* PCI multiple read/write */
mmio_write32(TxLow,  (uint32_t)tx_ring_pa);
mmio_write32(TxHigh, (uint32_t)(tx_ring_pa >> 32));
mmio_write32(RxLow,  (uint32_t)rx_ring_pa);
mmio_write32(RxHigh, (uint32_t)(rx_ring_pa >> 32));
mmio_write8 (Cfg9346, 0x00);                /* Lock */
mmio_write16(IntrMask, 0);                   /* polling, no IRQs */
mmio_write16(IntrStatus, 0xFFFF);            /* clear any pending */
mmio_write8 (ChipCmd, CmdRxEnb | CmdTxEnb);  /* go */
```

After this, register netdev with `name="eth0"`, `mtu=1500`, `send`/`poll` callbacks, then announce:
```c
printk("[NET] OK: rtl8169 eth0 mac=%x:%x:%x:%x:%x:%x\n", ...);
```

**Test:** Boot, see `[NET] OK: rtl8169 eth0 ...`. DHCP will fire next at vigil startup; expected behavior is "DHCP DISCOVER timeout" because there's no DHCP server in the test environment. The point of Task 5 is just `[NET] OK`.

## Task 6: Send path

```c
static int rtl8169_send(netdev_t *dev, const void *pkt, uint16_t len) {
    uint16_t slot = tx_head & 0xFF;
    /* Wait if device still owns this slot */
    if (tx_desc[slot].opts1 & DescOwn) return -1;  /* ring full */

    memcpy(tx_buf_va[slot], pkt, len);
    uint32_t opts1 = DescOwn | FirstFrag | LastFrag | len;
    if (slot == 255) opts1 |= RingEnd;
    tx_desc[slot].opts2 = 0;
    tx_desc[slot].opts1 = opts1;  /* DescOwn last */
    arch_wmb();
    mmio_write8(TxPoll, 0x40);    /* NPQ — kick normal-priority TX */
    tx_head++;
    return 0;
}
```

The `arch_wmb()` ensures the descriptor write is globally visible before the doorbell write. **Important:** `opts1` (containing `DescOwn`) MUST be written last so the chip doesn't race a half-initialized descriptor.

If link is up, `tcpdump -i enp2s0 arp` on the host (after re-binding 02:00.0 back to r8169 temporarily, OR via VFIO sniff which we don't have) should... well, with VFIO passthrough the host's r8169 isn't bound, so we need a different verification path:

## Task 7: Poll/RX path

```c
static void rtl8169_poll(netdev_t *dev) {
    while (1) {
        uint16_t slot = rx_head & 0xFF;
        uint32_t opts1 = rx_desc[slot].opts1;
        if (opts1 & DescOwn) break;  /* device still owns it — no more frames */

        uint16_t rlen = (uint16_t)(opts1 & 0x3FFF);
        if (rlen >= 14)  /* at least Ethernet header */
            netdev_rx_deliver(dev, rx_buf_va[slot], (uint16_t)(rlen - 4)); /* strip CRC */

        /* Hand the descriptor back to the device */
        uint32_t new_opts1 = DescOwn | RX_BUF_SIZE;
        if (slot == 255) new_opts1 |= RingEnd;
        rx_desc[slot].opts2 = 0;
        rx_desc[slot].opts1 = new_opts1;
        arch_wmb();
        rx_head++;
    }
}
```

Note: RTL strips the trailing 4-byte CRC by default? Actually, opts1 length on RX **includes** the FCS by default. We strip the trailing 4 bytes when delivering.

**Test:** With cable plugged in, run `[ARP] who-has 10.0.0.1` → expect to see `[ARP] reply` from the gateway. If gateway responds, the round trip works.

Then full DHCP. Vigil starts dhcp service automatically. Expected: vigil's dhcp runs, `[DHCP] OFFER` / `[DHCP] ACK` lines appear in serial, eth0 gets an IP.

Then ICMP self-test from `net_init`. Then a manual ping from inside Aegis to the gateway.

## Open risks

1. **Power state.** If the device is genuinely stuck in D3 and the PMCSR write doesn't cover it, MMIO reads will return 0xFFFFFFFF. Mitigation: log post-mapping `mmio_read32(MAC0)` early — if it's `~0`, we know mapping or power is wrong, fail loudly.
2. **PHY state.** If link doesn't come up under VFIO, the PHY isn't auto-negotiating. Most likely cause: r8169 left ALDPS enabled before we unbound. Mitigation: poll PHYstatus (0x6c) after init; if bit 1 (LinkSts) is 0, log `[RTL8169] WARN: link down` and continue (driver still works for TX, just nothing comes back). PHY init magic is a Phase 49b problem if needed.
3. **Endianness.** `opts1`/`addr` are little-endian. x86_64 is little-endian, so direct stores work.
4. **DMA address ceiling.** RTL8168 supports 64-bit DMA addresses (we set both *High registers). Aegis pmm pages can be above 4GB on machines with >4GB RAM. Both rings and bounce buffers come from `kva_alloc_pages` which calls `pmm_alloc_page`. If we hit a 32-bit-only chip variant we'd need a bounce-below-4GB allocator, but the 8168 spec sheet says full 64-bit support — fine.
5. **Bus master enable timing.** Some chips ignore Bus Master writes if Memory wasn't enabled in the same command-register write. We set both bits in one write to be safe.
6. **MSI/MSI-X.** RTL chip exposes MSI and MSI-X capabilities. We DON'T enable either — we leave them disabled and rely on polling. The VFIO container exposes them but unused is fine.
